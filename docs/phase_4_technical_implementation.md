# Phase 4 Technical Implementation

## Scope of This Technical Record

This document captures implementation details for Phase 4 (raw ingestion into DuckDB):
- what was implemented
- how it was implemented
- commands/scripts/files used
- validation outcomes

## What Was Implemented

1. Raw ingestion script:
- `src/ingest_raw_duckdb.py`
- loads each source CSV into DuckDB `raw` schema
- preserves raw content (line-by-line) without normalization

2. Raw validation script:
- `src/validate_raw_duckdb.py`
- validates table existence, row counts, and encoding integrity

3. Generated phase outputs:
- `data/reports/raw_ingestion_log.csv`
- `data/reports/raw_load_validation_report.md`
- DuckDB raw tables in `data/processed/regional_resilience.duckdb`

## How It Was Implemented

### Ingestion Design

1. Use `iso-8859-1` decoding explicitly for all source files.
2. Read each file as raw text lines (`splitlines`) to avoid accidental cleaning.
3. Split lines by `;` only to populate column slots (`col_1..col_n`) while also storing full `raw_line`.
4. Create schema `raw` and one table per source file:
- `raw.raw_22541_01_01_4`
- `raw.raw_22542_01_02_4`
- `raw.raw_23111_01_04_4`
5. Log ingestion metadata to CSV:
- file name, timestamp, encoding, delimiter
- source row count and max column count
- metadata/data-like/footer row counts
- loaded row count and ingestion error

### Validation Design

1. Verify each raw table exists.
2. Compare source line counts to loaded row counts (exact parity).
3. Verify encoding integrity:
- no replacement character rows (`�`)
- non-ASCII source lines match stored `raw_line` exactly by line number
4. Emit markdown summary report with pass/fail status.
